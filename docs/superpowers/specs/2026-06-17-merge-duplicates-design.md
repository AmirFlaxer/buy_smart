# עיצוב: מיזוג פריטים כפולים בין משתמשים

**תאריך:** 2026-06-17
**סטטוס:** מאושר - מוכן לתכנון מימוש

## מטרה
כששני משתמשים מוסיפים את אותו פריט לרשימה משותפת (למשל שניהם מוסיפים "חלב"), נוצרות שתי שורות. מנגנון מניעת הכפילויות הקיים (`HomeViewModel.quickAdd`) רץ רק בצד הלקוח ברגע ההוספה, ולכן מפספס מצב race שבו שני המשתמשים מוסיפים לפני שהסנכרון הפגיש את השורות. פיצ'ר זה מזהה כפילויות שכבר נוצרו, מדגיש אותן, ומציע מיזוג ידני לשורה אחת.

## החלטות מוצר (מאושרות)
- **התנהגות:** סימון והצעה - לא מיזוג אוטומטי. הדגשה ויזואלית על השורות הכפולות + כפתור "מזג".
- **זיהוי כפילות:** שם מנורמל זהה בלבד (לא מתחשב בהערה/מיקום). מתחשב באילו פריטים פעילים (לא-נקנו, לא-pendingRefill).
- **כמות במיזוג:** הגדולה (max), לא סכום. ההיגיון: שני המשתמשים מתארים את אותו צורך, וההערכה הגבוהה היא הבטוחה.
- **יחידות שונות (משקל מול ספירה):** לפי הגדרה חדשה בהגדרות המשתמש - ברירת מחדל: משקל גובר.
- **הערות במיזוג:** משולבות (לא נמחקות) כדי לא לאבד מידע.
- **מיקום התצוגה:** מסך הבית (HomeScreen) בלבד. מסך הקנייה לא מושפע.

## גישה (מאושרת)
זיהוי בצד הלקוח בזמן תצוגה: ה-ViewModel מקבץ פריטים לפי שם מנורמל בכל snapshot ומסמן קבוצות כפולות. המיזוג מתבצע בכתיבת batch אטומית. פותר את ה-race כי הזיהוי קורה אחרי שהסנכרון הפגיש את השורות, ועובד בלי שרת (תואם לאילוץ החינמי - Spark, ללא Cloud Functions).

חלופות שנדחו:
- **Cloud Function** - דורש Blaze בתשלום.
- **שם כ-document ID** - מונע כפילות במקור אך מתנגש בבחירת "סימון והצעה", ובעייתי עם שמות עבריים כמזהה.

## רכיבי לוגיקה טהורים (domain/util - כולם testable)

### ItemNameKey
נרמול שם פריט למפתח השוואה: `trim` + `lowercase` + צמצום רווחים פנימיים כפולים לרווח יחיד. `"חלב "` ו-`"חלב"` נותנים אותו מפתח. (זהה בדפוסו ל-`saveHistory` הקיים שמשתמש ב-`trim().lowercase()`.)

### QuantityMerge
הרכיב הרגיש ביותר. ממזג שתי מחרוזות כמות בהינתן העדפת יחידה (`WEIGHT`/`COUNT`):
- מנתח כל כמות ל-(מספר, יחידה טקסטואלית, סוג: WEIGHT / COUNT / UNKNOWN). זיהוי משקל/נפח לפי מילון יחידות עברי: ק"ג, קג, קילו, גרם, גר, ג', ליטר, ל', מ"ל, מל. כל השאר (ריק, "יחידות", "חבילה", מספר בלבד) = COUNT.
- **אותו סוג** (שתיהן WEIGHT או שתיהן COUNT): נרמול לבסיס משותף (גרם / מ"ל / יחידה) להשוואה, ובחירת הגדולה בייצוג המקורי שלה. כך "1.5 ק"ג" מנצח "500 גרם".
- **סוג שונה** (WEIGHT מול COUNT): נבחרת הכמות מהסוג המועדף לפי ההגדרה.
- **מקרי קצה** (לא ניתן לנתח מספר): best-effort - המחרוזת הראשונה הלא-ריקה.

### mergeItems
בהינתן קבוצת פריטים (אותו שם מנורמל) → פריט ממוזג אחד + רשימת IDs למחיקה:
- **הפריט השורד:** הראשון בקבוצה (מקבל את הערכים הממוזגים). השאר נמחקים.
- **כמות:** דרך `QuantityMerge` (מצטבר על פני הקבוצה).
- **הערות:** שילוב ייחודי - פיצול כל ההערות על `", "`, `distinct`, איחוד מחדש. "1%" + "3%" → "1%, 3%".
- **עדיפות:** הגבוהה ביותר בקבוצה (URGENT > NORMAL > NOT_URGENT, כלומר `minByOrNull { priority.ordinal }`).
- **סוג:** אם פריט כלשהו RECURRING → RECURRING, אחרת ONE_TIME.
- **תמונה:** ה-`imageUrl` הראשון הלא-ריק.

## שכבת הנתונים
- **`FirestoreService.mergeItemsBatch(listId, survivor: ShoppingItem, deleteIds: List<String>)`** - WriteBatch אטומי: `update` של הפריט השורד (name, quantity, note, priority, type, imageUrl) + `delete` של כל מזהה ב-deleteIds. כשל רשת באמצע לא משאיר מצב חצי-ממוזג (אותו דפוס כמו `finishShoppingBatch`).
- **`ItemRepository.mergeDuplicates(group: List<ShoppingItem>, unitPreference: String)`** - מפעיל את `mergeItems` הטהור, קורא ל-`mergeItemsBatch`, ומעדכן את Room (מחיקת ה-IDs + עדכון השורד).

## ההגדרה (DataStore)
- מפתח חדש `merge_unit_preference`: `"WEIGHT"` (ברירת מחדל) / `"COUNT"`.
- חשיפה דרך `ListRepository` (כמו `active_list_id`): `getMergeUnitPreference(): Flow<String>` + `setMergeUnitPreference(value: String)`.
- ב-UI: `DropdownMenuItem` חדש בתפריט ה-overflow ב-TopAppBar ("העדפת מיזוג: משקל / יחידות"), מחליף בין הערכים.

## ViewModel (HomeViewModel)
- ב-`observeItems`, בנוסף לקיבוץ הקיים לפי `LocationKey`: קיבוץ הפריטים הפעילים לפי `ItemNameKey.of(name)`, סינון קבוצות עם `size > 1`.
- שדות חדשים ב-`HomeUiState`: `duplicateGroups: Map<String, List<ShoppingItem>>` (מפתח שם מנורמל → פריטי הקבוצה). אילו שמות מודגשים נגזר מ-`duplicateGroups.keys`.
- איסוף ההגדרה `merge_unit_preference` ל-state (לשימוש בעת מיזוג).
- פעולה `mergeDuplicates(nameKey: String)`: שולפת את הקבוצה מ-`duplicateGroups`, קוראת ל-`itemRepository.mergeDuplicates(group, unitPreference)`, עם `errorMessage` ב-catch (כמו שאר הפעולות).

## UI (HomeScreen)
- שורת פריט ששייכת לקבוצת כפילות (השם שלה ב-`duplicateGroups.keys`) מקבלת **הדגשה**: רקע/מסגרת בגוון `md_errorContainer` (קיים ב-Color.kt) + אייקון אזהרה קטן.
- ליד הקבוצה המודגשת - כפתור קומפקטי "מזג" שקורא `mergeDuplicates(nameKey)`. הלחיצה ממזגת את כל הקבוצה לשורה אחת.
- ההדגשה נעלמת אוטומטית כשהמיזוג הושלם (ה-snapshot הבא לא מכיל כפילות).

## בדיקות
- **Unit tests (הליבה):**
  - `ItemNameKey`: trim, lowercase, רווחים כפולים.
  - `QuantityMerge`: אותה יחידה (max), ק"ג מול גרם (נרמול), משקל מול ספירה לפי העדפה (שני הכיוונים), מחרוזת לא-parseable (fallback).
  - `mergeItems`: שילוב הערות, בחירת עדיפות גבוהה, סוג RECURRING גובר, רשימת IDs למחיקה נכונה, בחירת תמונה.
- **בדיקה ידנית:** שני מכשירים מוסיפים "חלב" → הדגשה מופיעה בשניהם → לחיצה על "מזג" מאחדת לשורה אחת בשני המכשירים.

## מחוץ ל-scope (YAGNI)
- מיזוג אוטומטי ללא אישור משתמש.
- מיזוג בין מיקומים/קטגוריות שונים (זיהוי לפי שם בלבד).
- נרמול שמות חכם (מילים נרדפות, תיקון שגיאות כתיב).
- מיזוג במסך הקנייה.
